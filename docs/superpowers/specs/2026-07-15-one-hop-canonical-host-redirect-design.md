# One-Hop Canonical Host Redirect Design

**Date:** 2026-07-15

## Goal

Send every request for the bare `sisiwroclaw.pl` host directly to the matching
`https://www.sisiwroclaw.pl` URL in one permanent redirect while preserving the
path and query string.

## Confirmed production behavior

The canonical `www` site is live and indexable, but the bare HTTP host currently
uses two redirects:

1. `http://sisiwroclaw.pl/pl/` -> `https://sisiwroclaw.pl/pl/`
2. `https://sisiwroclaw.pl/pl/` -> `https://www.sisiwroclaw.pl/pl/`

Both responses are generated by Netlify. The repository currently contains only
path-level redirects, so no rule can intercept the bare HTTP request and send it
straight to the final host.

## Chosen approach

Add two forced domain-level rules at the beginning of `netlify.toml`, before the
existing path redirects:

```toml
[[redirects]]
  from = "http://sisiwroclaw.pl/*"
  to = "https://www.sisiwroclaw.pl/:splat"
  status = 301
  force = true

[[redirects]]
  from = "https://sisiwroclaw.pl/*"
  to = "https://www.sisiwroclaw.pl/:splat"
  status = 301
  force = true
```

Netlify documents separate HTTP and HTTPS rules for protocol-specific domain
redirects, requires the alias to be assigned to the site, processes the first
matching rule, and automatically forwards query strings for 301 redirects. The
current live HTTPS bare-host redirect proves the alias is already assigned.

## Alternatives considered

- Keep the two-hop chain. This avoids a change but retains avoidable latency and
  fails the audit's one-hop acceptance requirement.
- Add an Edge Function. This adds runtime code and operational surface for a
  redirect the Netlify rules engine supports natively.
- Change only a Netlify dashboard setting. The `www` host is already primary;
  the missing behavior is protocol-specific and is better kept in versioned
  repository configuration.

## Scope and safety constraints

- Do not add or change any page text, metadata copy, UI element, event, hours,
  image, structured-data fact, or placeholder.
- Do not change the final canonical origin, which remains exactly
  `https://www.sisiwroclaw.pl`.
- Preserve every requested path through `:splat`.
- Preserve query strings through Netlify's documented 301 behavior.
- Leave direct `www` requests untouched.
- Keep all existing locale and legacy path redirects after the domain rules.
- Preserve `.claude/worktrees/b2b/` and unrelated untracked files.

## Verification design

Add a regression test in `scripts/workflows.test.mjs` that reads `netlify.toml`
and asserts:

- the HTTP and HTTPS bare-host rules both exist exactly once;
- both target the final `www` host with `:splat`;
- both are forced permanent redirects;
- both appear before the existing root-to-Polish redirect;
- no rule redirects the final `www` host.

Run the focused test red before editing `netlify.toml`, then green after the
minimal configuration change. Run the complete Node 22.12 release gate before
publication. Publish through the repository's temporary validation branch and
required GitHub check, then fast-forward the exact validated SHA to `main`.

Production acceptance requires:

- `http://sisiwroclaw.pl/pl/?utm_source=redirect-test` returns one `301` directly
  to `https://www.sisiwroclaw.pl/pl/?utm_source=redirect-test`;
- `https://sisiwroclaw.pl/pl/?utm_source=redirect-test` returns the same direct
  final location;
- `https://www.sisiwroclaw.pl/pl/` returns `200`;
- canonical metadata still uses `www`, canonical pages still omit robots meta,
  and the complete live-host smoke test passes.

If the deployed Netlify edge does not honor the documented HTTP domain rule,
stop after preserving the response evidence. Do not add an Edge Function or
change DNS/domain settings without a separate design.

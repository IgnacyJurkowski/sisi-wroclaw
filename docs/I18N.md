# Internationalization (i18n)

The site is statically generated in five languages. All critical content is in
the rendered HTML (no client-only translation).

## Supported locales

| Code | Language | OG locale |
|------|----------|-----------|
| `pl` | Polski (source / fallback) | `pl_PL` |
| `en` | English | `en_GB` |
| `de` | Deutsch | `de_DE` |
| `it` | Italiano | `it_IT` |
| `cs` | ÄŚeĹˇtina | `cs_CZ` |

Defined in `src/i18n/config.ts`. Polish is the source language; any untranslated
key falls back to Polish and triggers a dev-time `console.warn` (see
`src/i18n/t.ts`). In production the typed dictionaries make a missing key a build
error, so production pages are fully translated.

## Architecture

- **Routes** â€” one catch-all (`src/pages/[...path].astro`) generates every
  `(locale, route)` page from the central route map in `src/i18n/routes.ts`.
  URLs are locale-prefixed with translated slugs (e.g. `/de/veranstaltungen/`).
- **Components are shared**, content is locale-specific. Page bodies live in
  `src/components/pages/*`; they receive a `locale` prop and read strings from a
  dictionary. There is no per-language copy of a component.
- **Dictionaries** â€” `src/i18n/ui/<locale>.ts`. `pl.ts` is canonical; its shape
  (`type UI = typeof pl`) is the contract every other locale must satisfy.
- **Legal content** â€” `src/i18n/legal.ts` (kept separate so it can fall back).
- **Formatting** â€” `src/i18n/format.ts` renders dates/numbers via `Intl` in
  `Europe/Warsaw`. Event times are stored as full ISO strings in
  `src/data/site.ts`; never hand-write weekday names.
- **Root + redirects** â€” `/` and all legacy Polish URLs 301 to their localized
  equivalents (`netlify.toml`).
- **SEO** â€” `Base.astro` emits per-locale `<html lang>`, canonical, full
  `hreflang` set + `x-default`, OG locale, and localized JSON-LD. The sitemap is
  generated at build (`src/pages/sitemap.xml.ts`) with hreflang alternates.

## How to update a translation

Edit the relevant key in `src/i18n/ui/<locale>.ts`. Keys mirror `pl.ts` exactly.
Brand-invariant data (prices, phone, cocktail names) lives in
`src/data/site.ts`, not the dictionaries.

## How to add a new locale

1. Add the code to `LOCALES` in `src/i18n/config.ts` and fill `LOCALE_NAMES`,
   `LOCALE_LANG`, `LOCALE_OG`.
2. Add a slug for every route in `SLUGS` (`src/i18n/routes.ts`).
3. Create `src/i18n/ui/<locale>.ts` (copy `pl.ts`, translate; `: UI` will flag
   anything missing) and register it in `src/i18n/ui/index.ts`.
4. Optionally add legal translations in `src/i18n/legal.ts` (otherwise it falls
   back to Polish under the convenience banner).
5. Run `npm test`.

## Type checking

`astro build` transpiles without type-checking `.astro` files. To validate the
typed dictionaries:

```
npx -y --package typescript@5.7 -- tsc --noEmit --strict --skipLibCheck \
  --module esnext --moduleResolution bundler --target es2022 \
  src/i18n/ui/index.ts src/data/site.ts
```

(Or add `@astrojs/check` + a `astro check` script for full `.astro` checking.)
